Case Number Subject Description
GCORE-1097 Return of a borrowed license does not work when FlexNet listening port is changed If a licensing administrator changes the FlexNet Manager listening port from TCP port 27000 to another port number (e.g. 4242), this will cause problems for Quantix-LA users when returning a borrowed license. The problem occurs because the new port number is missing in the command generated by the Quantix-LA GUI.

GCORE-166 7994: The Quantix license daemon is not creating the directories it needs in all cases. On Linux, the Quantix license daemon needs a .examinator dir in the $GEX_SERVER_PROFILE for linux platforms. this dir should be automatically created by the tool. The result of having no .examinator dir is on another system where there is a ym_admin_db and a gexdb, no matter how many times you start ym or pm, you cannot connect to the ym_admin_db. Also if you start with the -W command and see that you connect but on restart, you can no longer connect. Need to find portable functions that can create directories. This is a legacy issue. This is a configuration file that is needed and generated for legacy reasons. If directory does not exist, create it.

GEXYM-204 7851: Can't restrict Worldwide license to a user or group of users This issue affects large companies with multiple groups buying the same feature but with different options such as non WW and WW. Assuming we have 1 GEXPRO license non WW (entitlement A), and 1 GEXPRO WW (entitlement B). Assuming also that options have been set up in the license file so that group 1 only get access to entitlement A and group 2 only get access to entitlement B. Currently the Flex options do not allow to do so (specially EXCLUDE_ENTITLEMENT which is not intended to restrict the usage but the borrowing). We need to find a way to do it, either by differentiating the same products with different options with different expiration dates for instance. The problem is that even with the different expiration dates, it appears that EXCLUDE feature:expdate=<timestamp> does not work neither
