V3.12  (29 Jul 2016)
  • GCORE-8797: Continue processing remainder of split lot if corrupt wafer file is found 
    When a split lot is processed and one of the wafer files is corrupted, continue processing of the split lot and generate trigger files for all valid wafer files instead of stopping the process with no trigger file generated.

V3.10  (09 Nov 2015)
  • GCORE-5450: Update Quantix PAT triggering to handle new yyyyMMdd.hhmmss file format 
    The month code in the new yyyyMMdd.hhmmss format is a two-digit code so that files with this time stamp can be sorted and organized in chronological time stamp order.
V3.9  (18 Aug 2015)
  • GCORE-4783: G-Trigger doesn't generate summary log files anymore 
    G-Trigger does not generate any summary log file anymore.
V3.8  (12 Aug 2015)
  • GCORE-4754: G-Trigger is incorrectly generating summary logs 
    When determining if the Eagle PAT Log Files have or have not been processed into the summary log, G-Trigger uses the time stamp in the file name to ensure the latest data is reflected in the summary log. In v3.3 of G-Trigger, the format of that time stamp changed from HHMMSS_YYYYMMDD to YYYYMMDD_HHMMSS. This format change is now reflected in the reading of the PAT logs.
V3.7  (03 Jul 2015)

V3.6  (15 Apr 2015)

V3.5  (22 Dec 2014)
  • GCORE-2004: G-Trigger update to accomodate JSON recipes 
    G-Trigger now first looks for the JSON version of any recipe. If not found, it looks for the CSV version.

V3.4  (02 Aug 2014)
  • GCORE-1301: G-Trigger lot summary is incorrect

V3.3  (23 May 2014)
  • GCORE-794: Modify the way G-Trigger creates GTF file names for Eagle flow 
    G-Trigger used to generate Eagle GTF files using the STDF file name and replacing the
    .std extension with .gtf. The timestamp included in the STDF file name is not appropriate
    for alphabetic file ordering used in GTF trigger pumps. G-Trigger was enhanced to modify the
    timestamp used in the GTF file name (extracted from the STDF file name) so it is more appropriate 
    for file ordering (yyyyMMdd_hhmmss). 
    The codes were also updated to process summary files (FetTest, Eagle, Spektra) oldest to newest.

V3.1  (29 November 2013)
  • Case 7783: PAT and NoPAT wafer mix 
    G-trigger now supports splitlots with mixed PAT and NoPAT wafers.
    If a splitlot has PAT and NoPAT wafers, the GTF files will be created accordingly for each wafer.
    For a same wafer, if there are PAT and NoPAT files, only the PAT files are considered.
    Also, only PAT files will be used if a composite map is generated.

V2.5  (23 October 2013)
  • Case 7409: G-Trigger Update for ZPAT 
    add the ability for G-Trigger to distinguish the need for ZPAT from the PAT recipe and create a composite map
    trigger file and update the current trigger file generator to use the composite map where appropriate.
    The following 2 new variables are supported in the g-trigger.ini file:
    • Section=Options, Field=Output_Pat_Composite_Maps_Subfolder:
      subfolder to be used to generate the commosite map, the COMPOSITE_PAT GTF file, and the subfolder
      containing all individual wafer GTF files. If this option is not present, 'zpat' will be used as default
      subfolder. The subfolder gets created in the folder where the GTF files normally get created.
    • Section=Options, Field=Shell_Post_Composite:
      script to be used as value of the 'Shell' field in the COMPOSITE_PAT GTF files, so that it gets called at
      the end of the processing of a COMPOSITE_PAT GTF file.
  • Case 7693: Merge test and retest for PAT processing in the GTF files from g-trigger 
    when two files are present for one wafer within one lot.sublot, create only 1 gtf file with both the initial
    test and retest files included so that pm will do a file merge prior to pat processing.
    The following new variable is supported in the g-trigger.ini file:
    • Section=Options, Field=MergeTestRetest:
      if set to 1, one single GTF will be generated for each wafer, and test and retest wafer files for a same
      wafer will be listed in the same GTF, so that they will get merged during PAT processing.
      If set to 0, one GTF will be generated for each file, whether test or retest.

V1.21  (10 June 2009)
  • Rejected and delayed files management: 
    the way delayed and rejected files are treated has been modified. A delayed file <file> will be renamed to <file>.delay, and a <file>.log file is generated with the error message. A rejected file <file> will be renamed to <file>.bad, and a <file>.log file is generated with the error message. Also, the following 2 new .INI variables are supported:
    • [Options]
      ...
      Delayed_Files_Subfolder=<folder>

      if this variable is defined, the delayed file <file> is moved to this sub-folder instead of being renamed with the .delay extension, and the <file>.log file is also created in this sub-folder.

    • [Options]
      ...
      Rejected_Files_Subfolder=<folder>

      if this variable is defined, the rejected file <file> is moved to this sub-folder instead of being renamed with the .bad extension, and the <file>.log file is also created in this sub-folder.
